# Academic Project Report

## AI-Powered Online Examination System with a Real-Time Proctoring Module

This report is based strictly on the current implementation present in the repository. The codebase currently implements a substantial real-time AI proctoring backend, a telemetry and benchmarking layer, and a browser-based monitoring console. It does not yet implement a complete examination management stack such as authentication, question authoring, question delivery, submission storage, or automated academic grading. Accordingly, the phrase "online examination system" in this report refers to the intended broader university project, while all technical analysis of the current state is restricted to the modules that actually exist in code.

## 1. Introduction

Online education has evolved from a supplementary academic mechanism into a primary mode of instruction for schools, universities, professional certification bodies, and corporate training ecosystems. The expansion of remote teaching has accelerated the demand for digital assessment systems that can operate without the physical constraints of a traditional classroom or examination hall. In contemporary educational environments, institutions increasingly require platforms that can administer examinations at scale, allow geographically distributed participation, and provide a record of performance that can be integrated into larger learning workflows. This transition has elevated online examination systems from simple web forms into critical academic infrastructure.

The increasing use of online learning platforms has also exposed a structural weakness in conventional remote assessment methods. In a physical examination environment, integrity is supported by human invigilation, controlled seating arrangements, and restricted access to external resources. In a remote examination scenario, these constraints are weakened. Candidates may consult mobile phones, communicate with others outside the camera frame, move away from the device, or interact with multiple screens and external aids. Standard web-based quiz tools often assume honest participation and therefore offer limited mechanisms for verifying whether the observed user behavior corresponds to legitimate examination conduct.

The problem is not simply one of cheating in a narrow sense. It also concerns trust, fairness, and institutional credibility. A digital examination system that cannot provide a convincing account of candidate behavior weakens the value of the examination outcome itself. This is especially important in higher education, where assessments influence progression, certification, employability, and professional qualification. If remote assessment is to be accepted as academically equivalent to supervised in-person evaluation, the software infrastructure must support not only content delivery but also integrity assurance, runtime reliability, and explainable system behavior.

Conventional online testing tools such as static quiz systems, browser-based forms, or minimally instrumented learning management system assessments usually focus on question distribution and answer collection. They are often effective for low-stakes quizzes, practice assignments, or formative assessment, but they struggle in high-stakes contexts. These tools typically lack behavioral analytics, have no concept of real-time candidate monitoring, offer little or no adaptive handling of hardware variability, and do not provide detailed runtime telemetry that would help developers or institutions understand how the monitoring system behaves under different device conditions. Even when manual invigilation is used through videoconferencing, the approach is labor-intensive, inconsistent, and difficult to scale.

For this reason, there is growing interest in AI-based proctoring and adaptive examination systems. AI-based proctoring uses computer vision and behavioral inference techniques to observe candidate presence, gaze patterns, head orientation, movement behavior, and suspicious object appearance, particularly the use of mobile phones or secondary people entering the frame. An adaptive examination system extends this idea by recognizing that remote users do not operate on identical hardware. A robust proctoring platform must therefore monitor not only the candidate but also its own computational state. It must dynamically adapt frame rates, inference frequency, and model execution strategy so that integrity monitoring remains usable on CPU-only machines as well as on systems with dedicated GPUs.

The present project should therefore be understood as a modular and extensible online examination platform under development, with the current repository concentrating on the most technically demanding subsystem: the integrated AI proctoring engine and monitoring console. The codebase demonstrates a class-based, package-oriented design typical of an Object-Oriented Design and software engineering university project. The repository organizes functionality into separate modules for hardware profiling, scheduling, face detection, face tracking, head pose estimation, eye tracking, object detection, calibration, risk scoring, telemetry collection, performance reporting, and frontend visualization. This modular organization is academically significant because it reflects clear separation of concerns, supports replacement of individual algorithms, and creates a path for future integration with broader examination workflows.

From an educational software engineering perspective, this project addresses a central contemporary challenge: how to construct a digital examination environment in which exam integrity is not assumed but actively modeled, monitored, and reported. The implemented code shows that the project is not merely a conceptual discussion of AI proctoring. It already contains a functional real-time pipeline capable of ingesting webcam frames through a WebSocket connection, analyzing them using multiple computer vision components, adapting its behavior to detected hardware capability, and returning structured monitoring results to a live browser console.

## 2. Objective of the Project

The project has both academic and engineering objectives. At the academic level, the system aims to demonstrate how software engineering principles, object-oriented modularization, asynchronous communication, and AI-assisted perception can be combined to create a modern digital assessment platform. At the practical level, the repository seeks to provide the core technological foundation for a secure online examination environment in which candidate activity can be observed and assessed in real time rather than only after submission.

### A. Core Objectives

The first core objective is to design a modular examination platform rather than a monolithic application. The repository already reflects this intention through clear package boundaries such as `hardware`, `pipeline`, `vision`, `tracking`, `models`, `utils`, and `frontend`. This modular structure allows each functional concern to evolve independently. For a university project, this is especially important because it demonstrates sound object-oriented decomposition, improves maintainability, and makes the system easier to extend into a full academic examination suite later.

The second core objective is to enable automated integrity monitoring during remote assessments. The implemented system does not merely stream video; it converts visual input into semantic indicators such as face count, head pose, eye gaze direction, movement magnitude, hand-to-face interaction, suspicious object presence, frame quality warnings, and aggregated risk level. The underlying engineering motivation is to reduce dependence on continuous human invigilation and to provide a more consistent, machine-assisted basis for flagging suspicious behavior.

The third core objective is to support automated analytic output rather than raw sensor capture alone. The backend emits structured JSON responses and telemetry metrics instead of only visual overlays. This design allows downstream systems, such as reporting tools, review dashboards, or future institutional analytics modules, to consume the monitoring results programmatically. The presence of a session performance reporter further shows that the project considers system observability as part of its core goal, not as an afterthought.

The fourth core objective is to build a scalable architectural foundation. Although the current repository is not yet a distributed production system, it already separates transport, inference, scheduling, rendering, and telemetry concerns in a way that can later support cloud execution, horizontal scaling, and service decomposition. The code therefore serves both as a working prototype and as an architectural skeleton for a more complete platform.

### B. Technical Objectives

One major technical objective is real-time computer vision monitoring. This is achieved through a pipeline that combines MediaPipe Face Mesh based facial landmark extraction, custom head pose estimation using `solvePnP`, iris-based gaze estimation, MediaPipe Hands based hand detection, and YOLOv8n-based object detection. The engineering motivation is to cover multiple forms of suspicious behavior through complementary signals rather than a single binary detector.

Another technical objective is hardware-adaptive inference. The project uses a `HardwareDetector` to classify the host into `CPU_MODE`, `MID_GPU_MODE`, or `HIGH_GPU_MODE`, and then configures a scheduler accordingly. This objective is essential because remote examination software must function across a wide variety of student laptops. A fixed inference policy that performs well on a powerful workstation may fail or become unusable on an entry-level machine. The adaptive design attempts to preserve system responsiveness while maintaining acceptable monitoring coverage.

A further technical objective is dynamic frame scheduling. The system does not process every frame identically. Instead, it uses frame stride, face detection intervals, object detection rate limiting, and tracking between detection frames to reduce computational cost. This is a deliberate engineering choice intended to balance analytical fidelity against limited processing resources. The goal is not maximum raw analysis density at all times, but sustained monitoring under realistic deployment conditions.

Another technical objective is CPU/GPU adaptive model execution. In the current implementation, GPU adaptation primarily affects YOLO object detection, which is executed on CUDA when available and stable, but automatically falls back to CPU if GPU inference becomes unreliable. The motivation is operational resilience. Examination monitoring software must continue functioning even when ideal hardware conditions are absent or when runtime device errors occur.

The final major technical objective is extensibility through strong module boundaries. The system wraps face detection, pose estimation, eye tracking, object detection, risk scoring, calibration, and telemetry into separate classes with well-defined roles. This is a classic object-oriented design goal. It allows individual modules to be replaced, tuned, or benchmarked without rewriting the entire application, which is particularly valuable in a research or capstone project environment where algorithms may evolve during experimentation.

## 3. Problem Statement

Existing online examination platforms often fail because they treat the examination event as a simple sequence of web interactions: display a question, accept an answer, and store a submission. Such a model is sufficient for low-risk assessments, but it is inadequate for serious invigilated examinations. The core problem is that answer collection alone does not establish exam integrity. An institution can know what answer was submitted without knowing whether the candidate remained present, looked away repeatedly, used a mobile phone, received assistance from another individual, or temporarily moved out of frame.

One major difficulty in remote examinations is the lack of persistent monitoring. Many web-based systems do not inspect the candidate environment at all. They assume that the browser session itself is evidence of candidate presence. In reality, a browser window provides no assurance that the registered user is continuously present in front of the device. Face disappearance, frame obstruction, heavy motion, or replacement of the candidate by another person are all possible under such conditions.

Cheating via secondary devices is another significant problem. A mobile phone can be used to search answers, communicate externally, photograph questions, or receive messages. From an engineering standpoint, detecting such behavior is difficult because a phone may appear only briefly, may be partially occluded, or may resemble other small rectangular objects. The current repository addresses this challenge directly by integrating YOLO object detection with contextual phone filtering and temporal stabilization.

A further issue is behavioral ambiguity. Looking away from the screen once may be harmless, but persistent extreme gaze, repeated horizontal head movement, prolonged face absence, or strong off-camera orientation can be suspicious when considered together. Conventional platforms generally lack this form of behavioral analytics. They operate on simple event rules such as tab switching or full-screen exit, and even those features are absent from the present repository, which emphasizes that computer vision based behavioral modeling is itself a distinct subsystem requiring considerable design effort.

Existing systems also tend to be static in their computational assumptions. They often ignore the fact that the student population uses heterogeneous devices. An online proctoring algorithm that assumes stable GPU inference on every machine is impractical. Conversely, a system designed only for the weakest hardware may become too conservative and underutilize available acceleration on stronger devices. Static processing pipelines therefore create a dual problem: instability on weak devices and inefficiency on strong ones.

Another structural limitation in typical quiz systems is the absence of extensible software architecture. A tightly coupled design makes it difficult to add new cheating detectors, change models, benchmark performance, or provide detailed developer telemetry. In contrast, the present system attempts to solve this through a package-based organization in which hardware detection, frame scheduling, visual inference, risk analysis, and frontend rendering are separated into explicit modules. This is not only a software design preference but a practical requirement for AI-enabled systems, where iteration over thresholds, models, and runtime heuristics is expected.

The proposed system, in its current implemented state, addresses these problems through four principal strategies. First, it introduces modular architecture so that perception, scheduling, telemetry, and user interface concerns are decoupled. Second, it employs AI proctoring through face analytics, gaze estimation, head pose estimation, hand-face interaction analysis, and phone detection. Third, it performs behavioral monitoring rather than single-event inspection, aggregating multiple indicators into a smoothed risk score. Fourth, it uses adaptive system design, adjusting frame processing and model execution according to detected hardware capability and runtime load feedback.

In summary, the problem addressed by this project is not merely how to digitize an examination, but how to produce a trustworthy remote assessment environment whose integrity mechanisms are computationally practical, behaviorally meaningful, and architecturally extensible.

## 4. Scope of the Project

The current scope of the implemented system is best described as a real-time AI proctoring and monitoring subsystem for a broader online examination platform. The repository supports live webcam-based observation, asynchronous frame ingestion over WebSockets, per-frame computer vision inference, behavioral risk scoring, adaptive scheduling based on hardware profile, runtime resource telemetry, session-level benchmarking, and a browser dashboard for monitoring these outputs. The system also includes an interactive calibration workflow that personalizes several thresholds according to the observed user and environment before monitoring begins.

Within this scope, the project currently supports real-time AI proctoring in a technically meaningful sense. It can detect faces, maintain tracked identities across frames, infer head orientation, estimate eye gaze direction and offset, detect hand proximity to the face, classify suspicious objects with an emphasis on phone-like objects, warn about low-light or blocked-view conditions, and aggregate these signals into a scored risk representation. The backend also supports in-memory risk snapshot capture when risk exceeds a configured threshold, giving the monitoring session a limited incident capture capability.

The system also supports a hardware-adaptive inference pipeline. At session start, the host machine is profiled using CPU core count, RAM size, and GPU availability and memory. Based on this profile, the system selects one of three runtime modes and configures frame stride, face detection interval, target frame height, and YOLO inference rate. During execution, the scheduler can further increase processing stride if CPU usage or frame latency indicates overload. This means the system scope extends beyond AI perception into adaptive runtime management.

The asynchronous processing pipeline is another major area within scope. The frontend continuously captures frames from a webcam using browser APIs, compresses them into JPEG data, and streams them to the FastAPI backend over a WebSocket connection. The backend then uses bounded queues and separate worker loops to ingest frames, process them, and send only the latest result back to the frontend. This demonstrates that the project scope explicitly includes transport engineering and runtime responsiveness, not only computer vision models in isolation.

Real-time telemetry monitoring is also within the present scope. The backend collects frame timing, frames per second, dropped frame counts, process and system CPU measurements, RAM consumption, GPU utilization and memory pressure, and a richer session report designed for cross-device comparison. The frontend displays a subset of these live metrics. Therefore, the system already functions as both a monitoring engine and a measurement platform for evaluating how that engine behaves on different laptops.

However, the scope is also bounded in important ways. The current repository does not implement large-scale deployment architecture such as distributed proctoring servers, message brokers, load balancers, or persistent multi-session orchestration. It runs as a single FastAPI process serving a static frontend and handling WebSocket sessions directly. While this is entirely appropriate for a prototype and academic submission, it means that the project should not yet be described as a production multi-tenant exam platform.

Institutional role management is also outside the current scope. There are no modules for student registration, authentication, invigilator dashboards with user management, instructor accounts, admin roles, or access control policies. Likewise, there is no database schema for exam metadata, scheduled assessments, candidate rosters, or audit logs. The present repository monitors a session but does not manage the academic lifecycle surrounding that session.

Adaptive exam difficulty, automatic question sequencing, answer evaluation, and result publication are similarly not part of the implemented code. Although these are natural features for a complete online examination platform, the current repository contains no question bank, no exam engine, no submission store, and no grading logic. This is an important implementation-faithful limitation: the repository currently realizes the integrity-monitoring layer of the envisioned system rather than the full instructional assessment platform.

Despite these boundaries, the present system has meaningful scalability potential. Its modular decomposition would support integration with institutional LMS components, deployment behind API gateways, storage of telemetry in external systems, replacement of specific AI models, and connection to incident review tools. Real-world applications include remote university examinations, certification practice systems, recruitment assessments, training compliance tests, and secure mock-test environments. In each of these cases, the currently implemented proctoring subsystem could act as the integrity and observability layer of a larger examination ecosystem.

## 6. Modules Description

The repository is organized around the idea that real-time proctoring should be decomposed into cooperating but independent modules. The central backend class, `ProctoringSession`, acts as the runtime orchestrator, but the actual work is delegated to specialized components for hardware detection, scheduling, tracking, pose estimation, gaze estimation, object detection, calibration, risk scoring, resource monitoring, and session reporting. This organization reflects object-oriented design principles such as encapsulation, separation of concern, and interface-level composability.

At the same time, the broader examination modules that one would expect in a full online examination system, namely authentication, exam access control, question bank management, exam delivery, answer storage, and grading, are not present in the current repository. In implementation terms, the existing codebase is a proctoring-centric subsystem rather than a complete academic assessment platform. This is not a weakness in reporting; it is a necessary implementation distinction. The detailed module descriptions below therefore focus on the modules that truly exist in code.

### 6.1 AI Proctoring Module

The AI proctoring module is the conceptual center of the current system. Its runtime begins on the frontend, where the browser acquires webcam access using `getUserMedia`, selects an input camera, and captures video frames onto an off-screen canvas. The frontend intentionally reduces the transmitted processing size to a maximum width of 640 pixels and sends frames approximately every 90 milliseconds as compressed JPEG data over a WebSocket connection. Alongside each frame, the client sends contextual metadata such as viewport dimensions, screen dimensions, device pixel ratio, capture dimensions, and actual video dimensions. This client context becomes important later for adaptive gaze interpretation.

On the backend, the FastAPI application exposes a `/ws` WebSocket endpoint. Each WebSocket connection creates a `ProctoringSession`, which immediately performs hardware detection and initializes all major inference components. The session constructs a scheduler, face detector, adaptive face tracker, head pose estimator, eye tracker, hand-face interaction detector, frame quality analyzer, YOLO object detector, movement analyzer, risk engine, calibration manager, rolling FPS counter, resource monitor, and optionally a performance reporter. This session-local aggregation is a strong design decision because it makes each monitoring session self-contained while still allowing certain heavyweight resources, such as the YOLO model, to be shared across sessions.

The backend proctoring logic processes each frame through a staged pipeline. The incoming base64 string is decoded into an OpenCV image. The frame is then resized again on the server side according to the scheduler's target height policy and a global maximum width limit. This creates a two-stage resolution policy: the browser already downsamples to a practical transport size, and the backend can further constrain the frame according to detected hardware mode.

The first major analytic task is facial detection and tracking. Facial detection is implemented through a `FaceDetector` built on MediaPipe Face Mesh rather than a simple bounding-box-only detector. The detector extracts dense landmarks and then applies a series of geometric filters to reject implausible facial structures. These filters include area ratio limits, aspect ratio checks, minimum size requirements, relative placement checks for eyes, mouth, nose, chin, and forehead, and convex hull coverage checks. This indicates that the system is not content with accepting all raw landmark outputs from MediaPipe; it explicitly tries to suppress artifacts and non-face structures.

The face detector also performs temporal filtering. New candidate faces are not always accepted immediately. Instead, the detector maintains pending face hypotheses that must accumulate enough confirmation hits before being promoted to confirmed faces. This design reduces transient false positives, which are especially problematic in a proctoring context where a false second face could incorrectly imply collusion. The detector contains additional hardening logic for rapid horizontal movement, where background artifacts trailing the primary face can otherwise create spurious secondary detections. It also supports calibration-derived background motion zones, allowing regions with persistent non-face movement to be ignored during later monitoring.

Once detections are produced, identity persistence is managed by the `AdaptiveFaceTracker`. This module combines two strategies. During detection frames, a centroid-based assignment tracker gives persistent IDs to face detections. Between detection frames, lightweight OpenCV trackers such as KCF, CSRT, or MIL are used when available to update face boxes without rerunning full facial detection. When tracker updates fail, the module uses predictive hold logic based on previous velocity to briefly extrapolate face position. This hybrid strategy is central to the system's computational efficiency because it allows head pose, gaze, and movement analysis to continue even when full detection is temporarily skipped.

Tracking between detection frames is one of the most important optimization decisions in the repository. The scheduler does not run face detection on every frame. Instead, the system may detect periodically and track in between. If the tracker temporarily loses all faces during a non-detection frame, the pipeline performs a recovery detection. This means the design is not a naive skip-only strategy. It attempts to save computation while retaining a recovery path when tracking confidence deteriorates.

Head pose estimation is implemented in a custom `HeadPoseEstimator`. It selects six canonical facial landmarks and combines them with a fixed three-dimensional facial model. Using OpenCV `solvePnP`, it estimates rotation and translation vectors, converts the result to Euler angles, and derives yaw, pitch, and roll. The roll estimate is further blended with a direct eye-line geometric measurement to reduce jitter. The estimator maintains per-face Kalman filter state for smoothing over time, which is appropriate for noisy frame-by-frame angle estimation and improves stability for later risk analysis.

Eye tracking is handled by a custom `EyeTracker` that uses MediaPipe iris and eyelid landmarks. Instead of a full deep gaze network, the implementation computes normalized iris positions within each eye and derives horizontal and vertical ratios. These ratios are smoothed with exponential averaging, compared against a per-face baseline, and corrected slightly using head pose so that moderate head rotation does not automatically appear as an eye-gaze shift. The result includes a categorical direction such as `LEFT`, `RIGHT`, `UP`, `DOWN`, or `CENTER`, as well as continuous horizontal and vertical offsets and a confidence value. This is a pragmatic and explainable gaze model that fits well within a real-time academic project context.

Hand-face interaction is detected using MediaPipe Hands. For each detected hand, the system computes a bounding box and measures overlap against expanded face boxes. If the overlap ratio exceeds a configurable threshold, the face is marked as `handOnFace`. The importance of this feature is that it captures a suspicious but subtle form of behavior that simpler proctoring systems often ignore. In the current implementation, this signal is passed to the risk engine as a behavioral factor.

The project also includes a frame quality subsystem. The `FrameQualityAnalyzer` examines grayscale statistics, contrast, Laplacian variance, dark and bright pixel ratios, edge density, and center-region entropy. Based on these heuristics, it issues warnings for low light, dirty camera, or blocked view. Notably, these warnings are separated from the behavioral risk score. This is a thoughtful design choice. Poor lighting or lens blur may compromise monitoring quality, but they are not necessarily evidence of cheating. The system therefore keeps environmental quality advisories distinct from integrity-related risk factors.

Object detection is performed through `YoloObjectDetector`, which loads a shared YOLOv8n model from `yolov8n.pt`. The detector normalizes multiple raw YOLO classes into a reduced semantic set, currently focusing mainly on `person` and `phone`. Importantly, the phone pipeline is more elaborate than a single label lookup. The detector treats several surrogate classes such as `remote`, `laptop`, `tv`, `mouse`, and `book` as possible phone-like cues because a partially occluded phone or a phone in unusual orientation may not always be classified as `cell phone`. It then filters these candidates using confidence thresholds, bounding-box geometry, proximity to face and person regions, near-camera heuristics, temporal anchor matching, and background-motion-zone rejection. When no phone is found in the primary pass, the detector can run a lower-confidence probe pass using class-restricted, augmented YOLO inference to search more aggressively for phone-like objects.

This object detection module also stabilizes phone detections over time. A phone streak counter and anchor bounding box are maintained so that repeated or spatially consistent detections become more trustworthy. This is exactly the kind of temporal smoothing required in a proctoring system where false phone detections would be highly disruptive. The module therefore demonstrates that the repository is not simply using off-the-shelf YOLO predictions directly; it wraps them in domain-specific logic tailored to the examination problem.

The behavior signals produced by the proctoring module are fused by the `RiskEngine`. The engine currently scores nine rule families: multiple faces, phone detected, face missing, face not facing camera, high movement, extreme gaze, vertical head motion, horizontal head motion, and hand on face. Each rule has a base severity, and repeated activation across consecutive frames increases its multiplier after a configurable starting point. The raw score is then smoothed first through exponential moving average and then through a moving average window before being mapped to `LOW`, `MEDIUM`, `HIGH`, or `CRITICAL` risk bands. This multi-stage smoothing is important because it prevents risk from oscillating excessively in response to single-frame noise while still allowing persistent suspicious behavior to accumulate.

The repository's calibration system makes the proctoring module more individualized. Calibration proceeds in four stages: neutral baseline, gaze scan, vertical head motion, and hand-face interaction. During these stages, the system collects samples of neutral movement, neutral gaze noise, eye visibility, face area ratio, display diagonal context, gaze offsets, vertical pitch change, and hand-face flags. From these samples, it computes personalized thresholds for movement, gaze, head motion, face orientation, and phone detection sensitivity. It also learns persistent background motion zones by accumulating a grid-based motion map outside detected face regions. These zones are then fed back into the face detector and object detector so that known background movers are less likely to trigger false detections later. This is a particularly strong feature in the current repository because it shows a shift from static heuristics toward user- and environment-aware adaptation.

Another critical implementation detail is that risk scoring is intentionally held at zero during calibration. Guided calibration involves deliberate eye and head movements that would otherwise look suspicious. The code therefore zeroes the risk breakdown while calibration is active, preventing the calibration session from contaminating the later risk baseline. This reflects careful thinking about the distinction between training-time behavior and monitoring-time behavior.

The hardware-adaptive dimension of the proctoring module is summarized in the following table.

| Runtime mode | Selection logic in current code | Frame stride | Face detection interval | Target height | Target YOLO FPS | Practical interpretation |
| --- | --- | --- | --- | --- | --- | --- |
| `CPU_MODE` | No CUDA GPU available, or explicit CPU fallback after GPU instability | 4 | 10 | 480 | 3.0 | Conservative mode that processes fewer frames, detects faces less often, and keeps object detection sparse |
| `MID_GPU_MODE` | CUDA available, but hardware does not meet the high-tier thresholds | 2 | 5 | 720 | 8.0 | Moderate acceleration profile for common mid-range GPU laptops |
| `HIGH_GPU_MODE` | CUDA available with at least 8 GB VRAM, at least 10 logical CPU cores, and at least 16 GB RAM | 1 | 5 | 1080 | 12.0 | Full-rate processing profile, though frontend capture currently still limits transmitted width to 640 |

The adaptive scheduler does not stop at initial mode selection. It monitors CPU system percentage and total frame latency. If CPU reaches at least 85 percent or frame latency reaches at least 260 milliseconds for repeated samples, it increments a dynamic stride boost, causing the effective frame stride to increase by up to two additional levels. If the system later recovers, with CPU at or below 62 percent and frame latency at or below 180 milliseconds for a sustained interval, the dynamic stride boost is reduced. This creates a feedback-driven scheduler rather than a one-time profile lookup.

Overall, the AI proctoring module is the most developed and technically mature part of the repository. It demonstrates not only model integration but also temporal logic, adaptive scheduling, calibration, contextual filtering, and structured risk aggregation.

### 6.2 Asynchronous Processing Pipeline

The system's asynchronous processing pipeline is split across the frontend and backend, with both sides contributing to non-blocking operation. On the frontend, the browser runs a timed send loop rather than a tight synchronous frame upload. Frames are only transmitted when the WebSocket is open and no previous frame is still considered in flight. This means the client already applies a coarse form of backpressure by avoiding unlimited outstanding frame submissions. The frontend then reschedules itself with `setTimeout`, which is sufficient for a browser-based prototype and easy to reason about in an academic context.

On the backend, the WebSocket handler creates two bounded `asyncio.Queue` objects: a frame queue for inbound frame payloads and a render queue for outbound processed results. An `inference_worker` consumes frames from the frame queue, calls the session's processing function, and publishes only the most recent result to the render queue. A separate `render_worker` consumes from the render queue and sends JSON responses back to the browser. Because both queues are bounded, older work is explicitly discarded when the system becomes saturated. This is a deliberate design choice. In real-time proctoring, the newest frame is usually more valuable than a stale backlog, so dropping old frames is preferable to allowing latency to grow unbounded.

Another important aspect of the pipeline is that the actual per-frame analysis runs in a thread via `asyncio.to_thread`. This prevents CPU-heavy OpenCV, MediaPipe, and YOLO operations from blocking the event loop. At the same time, `ProctoringSession` protects processing with an asynchronous lock so that a single session does not execute multiple heavy analysis passes in parallel and oversubscribe local resources. Thus, the implementation balances asynchronous I/O with controlled synchronous inference.

Frame dropping is explicitly tracked. If a new frame arrives while the frame queue is already full, the oldest queued frame is removed and the dropped-frame counter is incremented. Invalid frames that fail base64 decoding are also counted as drops. The system therefore distinguishes between processed frames, skipped inference frames, and dropped frames. This matters because "inference skipped" is an intentional scheduling decision, whereas "dropped frame" indicates backpressure or malformed input.

The render side follows a similar newest-only philosophy. If the render queue is full, the oldest pending render result is discarded so that UI publication does not lag behind inference. This keeps the frontend dashboard visually responsive. The consequence is that the UI may not display every intermediate result, but it will preferentially display the newest state, which is desirable for live monitoring.

The repository also contains additional pipeline abstractions that are not currently wired into the main WebSocket runtime. `FrameCaptureThread` provides a dedicated local camera capture thread using OpenCV. `ThreadedInferenceEngine` provides a threaded bounded-queue inference executor that even supports small batching. `LatestResultRenderer` provides a non-blocking latest-only output buffer. These classes show that the project has explored a reusable threaded pipeline architecture beyond the currently active asyncio-WebSocket path. In the current codebase, however, these classes function more as reusable infrastructure or architectural scaffolding than as active runtime components.

This distinction is academically important. The implemented asynchronous pipeline that actually runs today is the WebSocket-based queue architecture inside `main.py`. The thread-based capture and inference classes indicate a possible next step toward local-camera desktop execution or more reusable service composition, but they should not be described as active runtime behavior in the present system.

The resulting pipeline can therefore be summarized as a three-layer asynchronous design. The frontend captures and compresses frames without blocking the UI. The backend ingests frames into a bounded queue and offloads inference work to a worker loop backed by a thread. The render worker publishes only the latest result to the browser. This separation improves performance because capture, inference, and rendering are decoupled, and no stage is allowed to create unbounded delay for the others.

### 6.3 Model Execution Strategy

The model execution strategy in the current repository is adaptive but selective. It is adaptive because the system detects available hardware, configures scheduler parameters accordingly, and allows object detection to run on CUDA when possible. It is selective because not every model in the pipeline is GPU-accelerated. This is a crucial implementation detail that an academically faithful report must emphasize.

The hardware detection process uses PyTorch CUDA checks along with `psutil` hardware data to determine whether the device should operate in `CPU_MODE`, `MID_GPU_MODE`, or `HIGH_GPU_MODE`. The detected profile includes CPU core count, RAM size, GPU name, GPU memory capacity, frame stride, face detection interval, target height, target YOLO FPS, and a nominal batch size. The engineering meaning of this profile is that it acts as a runtime policy object controlling later scheduling choices.

In the active implementation, the module most clearly affected by GPU selection is the YOLO object detector. The `YoloObjectDetector` loads a shared Ultralytics YOLOv8n model and moves it to `cuda:0` if CUDA is available, otherwise to CPU. It also exposes `set_runtime_device`, allowing the backend to force CPU execution when necessary. During inference, the detector passes the current runtime device into `predict`. If CUDA-based prediction throws an exception, the detector automatically moves the model to CPU and retries. At the session level, this failure also triggers a broader backend fallback action that changes the hardware profile to CPU mode, resets the scheduler, reduces the target YOLO FPS, and emits a `GPU_FALLBACK` warning. This is a strong example of graceful degradation.

By contrast, the face detector and hand detector rely on MediaPipe and are not explicitly switched between CPU and GPU in the repository. The head pose estimator, movement analyzer, frame quality analyzer, risk engine, and calibration logic are primarily numerical or OpenCV-based modules that also execute on CPU. Therefore, the current GPU strategy is best described as partial acceleration rather than full GPU offloading of the entire pipeline.

This partial acceleration is still meaningful. Object detection is typically the most computationally expensive stage among the implemented modules, especially when using a convolutional model like YOLO. By moving this stage to GPU where available and reducing its execution frequency on weaker hardware, the system captures a large portion of the possible performance benefit without requiring every component to be GPU-aware. This is a practical design choice for a university project, especially when broader device compatibility is more important than maximum hardware utilization on a small number of machines.

The repository also embodies another model execution strategy: temporal reuse of previous results. The object detector enforces a minimum interval between YOLO runs based on the scheduler's target YOLO FPS. If a new call arrives before that interval has elapsed, the detector simply returns cached results. Similarly, when the scheduler skips a frame's heavy inference path, the session reuses the last known analytic outputs for gaze, pose, hand-face status, objects, and warnings while still updating tracked face positions through lightweight tracking. This is not model execution in the narrow sense of choosing CPU versus GPU, but it is an execution strategy that substantially reduces computational demand.

The current implementation also contains an unused signal about future execution strategy: `batch_size` exists in the hardware profile and `ThreadedInferenceEngine` can batch pending items briefly before running inference. However, the active WebSocket runtime does not currently use this batching facility. As a result, the present system still operates primarily in a frame-by-frame manner, albeit with bounded queues and rate limiting. This is an important example of the difference between implemented infrastructure and currently wired behavior.

From a support perspective, this model execution strategy is well aligned with low-end and mid-range devices. On CPU-only systems, the system reduces processing density and explicitly forces YOLO to CPU mode. On mid-range GPU systems, it raises throughput but still uses conservative scheduling values. On high-end systems, it allows per-frame processing and higher YOLO frequency, though the browser-side capture limit means that the raw input itself remains modest. Thus, the execution strategy is best described as compatibility-first and stability-first rather than raw-throughput-first.

### 6.4 Performance Telemetry System

The repository contains a notably rich telemetry subsystem, which is one of its strongest engineering features. Performance monitoring begins with the `RollingFps` utility, which calculates processed frames per second over a sliding timestamp window. This provides live throughput estimates for each session and is reported to the frontend.

More detailed telemetry is provided by `ResourceMonitor`. This module samples process-level and system-level CPU usage, logical core count, system memory utilization, used and available RAM, process RSS and VMS memory, and thread count. When CUDA is available, it additionally records GPU memory statistics from PyTorch such as free, used, allocated, reserved, and maximum allocated memory. It further attempts to call `nvidia-smi` to capture GPU utilization percentage, GPU memory utilization percentage, temperature, power draw, and a second view of GPU memory use. To reduce overhead, the `nvidia-smi` snapshot is cached for a short interval.

These resource snapshots are merged with per-frame pipeline metrics. The session measures face processing time, YOLO time, pose and eye analysis time, frame quality analysis time, risk engine time, total frame time, FPS, dropped frames, processed frames, frame index, whether inference was skipped, effective frame stride, and face detection interval. These metrics are not merely shown live; they are also passed back into the adaptive scheduler, which uses CPU percentage and total frame latency to decide whether to increase or decrease stride.

At a higher abstraction level, the `SessionPerformanceReporter` aggregates these live metrics into a report-ready session artifact. It constructs time series for numerous values, computes summary statistics such as average, minimum, median, 95th percentile, maximum, and delta, and builds an exportable JSON report. The report contains a session block, a device description, a comparison vector, a summarized performance view, observations, an event log, a timeline, and an `llmReady` helper section intended for downstream automated analysis. This suggests that the telemetry system is designed not only for human dashboard use but also for comparative benchmarking and possibly future machine-assisted reporting.

The comparison vector is especially useful from an academic benchmarking perspective. It includes average and p95 FPS, average and p95 frame time, average and peak process CPU percentage, peak system memory percentage, peak process RSS, average GPU utilization, peak GPU temperature, peak GPU memory utilization, and average FPS per CPU percent. This is precisely the kind of normalized information needed when comparing how the same monitoring scenario performs on different laptops.

The reporter also identifies events by threshold crossing. It records spikes in process CPU usage, system RAM pressure, GPU temperature, GPU memory pressure, low FPS, frame latency spikes, and dropped-frame increments. These events are classified by severity and category. The report then generates observations such as low average FPS, CPU saturation risk, GPU thermal pressure, or memory growth. This makes the telemetry system significantly richer than a dashboard that only shows instantaneous values.

Another important property is persistence. On request, or automatically when a WebSocket session closes, the performance reporter can export the full JSON report to the `reports` directory. In the current repository, the frontend does not yet expose buttons for these performance-report commands, but the backend already supports `performance_report_status`, `performance_report`, `performance_report_export`, and `performance_report_reset` over the WebSocket control channel. This means the telemetry architecture is ahead of the current UI surface.

The telemetry system therefore serves three purposes simultaneously. First, it supports live situational awareness during monitoring. Second, it supports adaptive runtime decisions through scheduler feedback. Third, it supports offline benchmarking and comparison across devices. This three-layer design is unusual in small academic projects and represents a significant strength of the current implementation.

### 6.5 Frontend Monitoring Interface

The frontend monitoring interface is implemented as a single static `index.html` file served directly by FastAPI. It does not use a frontend framework such as React, Vue, or Angular. Instead, it combines HTML, CSS, and vanilla JavaScript into a tightly integrated live console. From a systems perspective, this makes the frontend lightweight and easy to serve, while also making the code straightforward to inspect in a university project setting.

The interface includes a live video panel, an overlay canvas, status indicators for WebSocket connection, camera state, and YOLO device state, buttons for starting and stopping monitoring, a calibration trigger, a statistics reset button, a log area, a camera selector, and a camera-switch mechanism. It also includes cards for risk level, risk score, face count, FPS, and dropped frames. These UI elements directly mirror the backend's runtime outputs.

One of the most important frontend components is the overlay drawing system. The interface draws face bounding boxes, face IDs, gaze direction, movement magnitude, phone and person boxes, and a directional line derived from head pose. It also draws animated calibration guides for each calibration stage. This means the frontend is not only a dashboard but a visual explanation layer that helps users understand what the backend currently sees and why certain risk signals may be appearing.

The interface also contains a risk-driver panel. When the backend sends a formal risk breakdown, the UI displays active rules, their current points, streak values, and score math. If the backend breakdown is missing, the frontend can infer a simplified local breakdown from returned faces and objects. This fallback is useful for robustness, although the backend route currently does send risk breakdowns by default. It is worth noting that the frontend's inferred rule set is somewhat simpler than the backend's full rule set, which further reinforces that backend risk logic is the authoritative source.

The live charts for FPS and risk score are implemented using canvas drawing rather than an external charting library. This is a practical and lightweight choice. The frontend stores recent history arrays, draws axes and line graphs manually, and updates chart metadata such as current, minimum, and maximum values. The result is a self-contained monitoring interface with no dependency on a JavaScript framework or third-party chart engine.

The resource panel visualizes backend telemetry values including CPU system and process usage, core count, RAM usage, process RSS, GPU utilization, GPU memory usage, temperature, power, and simple efficiency ratios. This panel demonstrates a close coupling between backend observability and frontend presentation. Although the backend offers richer performance-report features than the frontend currently exposes, the live resource panel already provides useful operational transparency.

The interface also includes a dedicated risk snapshot area. When the backend captures a risk snapshot, the frontend stores the returned base64 image, displays it as a thumbnail, and opens it in a modal when selected. This is an effective feature for incident review because it preserves visual evidence of high-risk moments without requiring continuous full-session recording.

A final interface strength is the calibration coach overlay. The UI translates backend calibration state into stage labels, guided instructions, progress bars, and animated visual hints that tell the user how to perform each calibration step. This transforms calibration from a purely technical backend process into an interactive user workflow, improving usability and making the personalized thresholding feature practically accessible.

The main limitation of the frontend is that it is a monitoring console rather than a full student exam interface. It does not display questions, manage timers for academic sections, store answers, or provide role-based views for instructors and students. Thus, its current function is best described as a proctoring dashboard, not a complete examination portal.

## 7. Current Development Status

The current development status of the repository can be characterized as a strong functional prototype of a real-time AI proctoring subsystem with advanced runtime observability and a usable browser-based console. The backend architecture is operationally coherent. It exposes a health route, serves a frontend, accepts webcam frames over WebSockets, processes them through a multi-stage analytic pipeline, and returns structured monitoring results. The implemented components are not placeholders; they contain detailed logic for filtering, tracking, smoothing, thresholding, and runtime adaptation.

Several major capabilities are already successfully implemented at code level. These include a functioning proctoring pipeline, hardware-adaptive scheduling, multi-stage calibration and threshold personalization, hybrid detection-and-tracking for faces, head pose and gaze estimation, YOLO-based object detection with contextual phone filtering, GPU-to-CPU fallback handling, risk scoring with repeated-violation amplification, frame-quality warnings, live resource sampling, session performance reporting, exportable telemetry reports, and a visual frontend dashboard with logs, charts, overlays, camera selection, and snapshot review.

The codebase also reflects thoughtful engineering around false positive control. Secondary face detections are stabilized before being accepted, background motion zones are learned during calibration, phone detection is contextualized and temporally smoothed, face orientation risk is modeled relative to a baseline rather than with a single hard angle threshold, and risk is explicitly frozen during calibration. These details indicate that the system has moved beyond a naive proof of concept into a more refined prototype.

However, the repository also has clear current limitations. The most important limitation is scope: it does not yet implement the broader examination platform components implied by the project title. There is no authentication, no exam scheduling, no question bank, no answer submission flow, no grading engine, and no result storage. The implemented system should therefore be described as the intelligent proctoring and monitoring core of a future online examination system, not the complete final platform.

Another limitation concerns deployment maturity. The system currently operates as a single FastAPI application with an in-memory session model. There is no database persistence for incidents, no distributed coordination, no external queue infrastructure, and no long-term audit storage beyond exported performance reports. Risk snapshots are stored in memory during the session and transmitted directly to the frontend. This is perfectly acceptable for a university project prototype, but it constrains scalability and formal auditability.

There are also runtime limitations directly visible in the current implementation. Although the backend defines a `HIGH_GPU_MODE` with a 1080 target height, the browser frontend currently downsamples transmitted frames to a width of at most 640 pixels before sending them. This means that the strongest backend hardware profile cannot yet exploit genuinely high-resolution browser input under the default UI. Similarly, while the repository includes a threaded inference engine with nominal batching support, the active WebSocket path still processes frames one by one and does not use the profile's batch size field.

The current UI surface is slightly behind the backend in some areas. The backend supports performance-report status, retrieval, export, and reset commands, but the frontend does not provide controls to invoke them. The telemetry data shown live is therefore only part of what the backend can already produce. In the same way, the frontend's locally inferred fallback risk rules are less complete than the backend's current full rule set, though this only matters if the backend breakdown were unavailable.

Testing support is also limited. The repository includes a simple `test_ws.py` script that acts as a developer smoke test for WebSocket processing, but there is no formal automated test suite, no unit tests for risk logic, no integration tests for calibration, and no load tests for concurrent sessions. Consequently, while the code is quite detailed, its validation infrastructure is still at an early stage.

Overall, the present development status is that of a technically substantial academic prototype. The most difficult subsystem, namely real-time adaptive proctoring, has already been implemented in considerable depth. The next steps are less about inventing the monitoring core and more about expanding the surrounding examination platform, operational hardening, and systematic validation.

## 8. Future Development Possibilities

No explicit roadmap file is present in the repository. Therefore, the future development discussion in this section is an inference from the current architecture, module boundaries, and missing platform layers rather than a claim about a formally documented project plan. Even with that constraint, the current code strongly suggests several realistic extension paths.

The first major extension is to complete the broader examination platform around the existing proctoring core. The repository would need authentication, session authorization, exam scheduling, question bank management, answer capture, timed exam flow, submission locking, grading, and result storage. From an architectural standpoint, the current proctoring engine is already modular enough to be embedded as one service within such a platform. The main additional work would involve persistent data models, role-based APIs, and workflow orchestration.

A second extension is more advanced cheating detection. The current risk engine uses interpretable rule aggregation rather than learned sequence modeling. This is a reasonable and transparent first step, but future development could incorporate machine learning models trained on temporal behavior sequences. For example, rather than thresholding gaze or head motion independently, the system could learn multi-signal temporal patterns associated with suspicious consultation behavior, peer assistance, or deliberate face hiding. The existing structured outputs from head pose, gaze, movement, and object presence already provide a strong feature substrate for such models.

The eye-gaze subsystem could be extended significantly. The current implementation estimates gaze through landmark geometry and baseline-relative offsets. Future versions could add finer gaze classification, probabilistic confidence calibration, explicit fixation duration modeling, or deep gaze-estimation models that are more robust to glasses, varied illumination, and off-axis webcams. Because the existing calibration system already collects gaze dynamics and display context, the architecture is well positioned for more advanced gaze personalization.

Multi-person analysis is another important direction. The current system can detect and risk-score multiple faces, but the logic is intentionally conservative in order to reduce false positives. A future production-oriented system could distinguish between transient false secondary detections, background faces on posters or screens, real intrusions, and identity swaps. This might involve temporal multi-person tracking, face recognition under consented institutional policy, or dedicated classification of candidate versus bystander.

Mobile phone detection can also be improved further. The current system already uses YOLOv8n, surrogate classes, contextual filters, temporal streaking, and a probe mode. Future work could introduce a specialized mobile-device detector trained specifically on examination scenarios, including partially occluded phones, reflected screens, and handheld devices near the edge of frame. Since the current implementation already separates phone thresholds and debug metrics, the detector is architecturally prepared for more specialized models.

Audio monitoring is a natural future extension. At present, the browser requests video only and the backend processes no audio. In many remote cheating scenarios, whispered communication or environmental conversation can be relevant. A future system could capture microphone input with explicit user permission, compute voice activity or speaker diarization, and integrate audio anomalies into a multimodal risk engine. This would substantially expand the integrity coverage of the platform.

Browser-behavior monitoring is another missing but important area. The current repository does not track tab switching, full-screen violations, window focus changes, clipboard access, or browser-level exam restrictions. Future work could combine these browser events with the existing visual pipeline so that the system reasons jointly about UI behavior and physical behavior. This would move the project closer to a production remote examination environment.

The exam engine itself could evolve in adaptive directions. Once question delivery exists, the project could support adaptive difficulty, dynamic question sequencing, or behavior-aware intervention logic. For example, if telemetry indicates that the monitoring pipeline is overloaded on a weak device, the exam interface could automatically shift to a lower visual quality monitoring mode while preserving exam continuity. Similarly, if high-risk events accumulate, the system could trigger additional identity verification steps or notify a human invigilator.

Distributed deployment is another realistic extension. The modular separation between transport, inference, telemetry, and frontend already hints at a future in which frame ingestion, model inference, and reporting are decoupled into separate services. A production version could use dedicated inference workers, persistent queues, centralized telemetry storage, and scalable session routing. This would be necessary if the system were to support institution-wide simultaneous examinations.

The telemetry system itself offers further research possibilities. The current `SessionPerformanceReporter` already exports structured comparison vectors and threshold-based observations. Future development could add automatic cross-session benchmarking dashboards, anomaly detection over many devices, and institution-level capacity planning tools. The telemetry output could also be linked with model quality metrics to explore the trade-off between computational cost and monitoring accuracy under different hardware tiers.

Incident reporting and forensic review could also be expanded. At present, high-risk snapshots are captured in memory and surfaced in the UI. A fuller system could persist these snapshots with timestamps, associated rule states, device profile, and exam metadata in a secure audit store. Human invigilators could then review not just images but a synchronized timeline of events, risk contributions, and resource state. This would make the system more suitable for appeals, post-exam review, and formal academic audit procedures.

Cloud deployment is another logical next step. The current repository is local-first and uses a static frontend served by FastAPI. A future cloud-native version could containerize the backend, store performance reports and incidents in managed storage, use CDN-backed frontend hosting, and integrate identity providers and institutional databases. The current modular codebase would support such a transition more readily than a monolithic script would.

Model optimization for deployment is also important. The current system uses YOLOv8n and several CPU-bound analytical stages. Future work could include ONNX or TensorRT optimization, quantization, more aggressive asynchronous batching, or specialized lightweight models for CPU-only environments. Because the system already measures frame latency, dropped frames, and resource pressure, it has the instrumentation needed to evaluate such optimization efforts scientifically.

The frontend could be expanded into a multi-role interface. A student-facing exam view, an invigilator live-monitoring view, and an administrator report view could all be built on top of the existing console concepts. The current UI already demonstrates useful components such as overlays, charts, logs, and snapshot review. These could be generalized into role-based dashboards once the surrounding examination workflow exists.

Finally, the current codebase opens an important research direction in calibrated personalization. The calibration manager already adjusts thresholds for movement, gaze, head motion, and phone sensitivity while also learning persistent background motion zones. This could evolve into a richer personalization framework that adapts not only thresholds but also model confidence weighting, camera placement expectations, display-size normalization, and user-specific noise profiles. In a production-grade proctoring platform, such personalization may be crucial for reducing false positives across diverse users and environments.

Taken together, these future possibilities show that the current repository is an effective foundation rather than an isolated experiment. Its architecture is already aligned with many of the extensions required for a production-grade AI-assisted proctoring platform. What remains is the systematic expansion of platform features, the maturation of deployment and testing infrastructure, and the continued refinement of detection logic into more robust multimodal integrity analysis.

## Conclusion

The current repository represents a serious implementation of an AI-based proctoring subsystem for a larger online examination project. It combines modular backend architecture, WebSocket-based real-time streaming, hybrid detection-and-tracking strategies, CPU/GPU adaptive execution, calibration-driven personalization, structured risk analysis, and detailed telemetry reporting. While it does not yet implement full exam management functions such as authentication, question delivery, grading, or institutional role management, it already solves a technically demanding part of the remote examination problem at a level well suited to a university software engineering submission.

Its academic value lies in the fact that it does not treat AI as an isolated add-on. Instead, the code integrates AI perception with scheduling, systems monitoring, object-oriented decomposition, and frontend explainability. For that reason, the repository should be understood as a modular and extensible foundation for a secure digital assessment platform, with the current implementation centered on real-time proctoring and runtime adaptability. In this form, it already demonstrates substantial engineering depth and provides a credible basis for future expansion into a complete online examination system.
